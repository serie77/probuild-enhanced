# ProBuild Operations Platform (v2) — Camunda 8 Orchestration

**Module:** UFCFAF-30-3 Development of Information Systems Project · UWE Bristol
**Engine:** Camunda 8.8.11 Self-Managed (c8run) · **Java:** 21 · **Spring Boot:** 3.5.9

An orchestration-based, multi-actor process automation for **ProBuild Supplies Ltd** — a
builders' merchant offering retail sales, tool hire, and a loyalty Trade Card, integrated with
two external partners (FinTrust finance, FixPro maintenance).

> This is a **ground-up re-architecture** of the case study. Where a lane-based collaboration puts
> every capability in one monolithic pool, this platform models the business as a lean **Customer
> Journey orchestrator** that delegates back-office work to **independent capability processes**
> (call activities) and reaches partners only through **correlated messages**. Business rules live
> in **DMN decision tables**. See [ARCHITECTURE.md](ARCHITECTURE.md) for the full design rationale.

---

## 1. What it does

| Journey | Flow |
|---|---|
| **Apply for Trade Card** | Choose service → apply → `Cap_Loyalty` (issue/renew + DMN band) → confirm |
| **Buy** | Browse → `Cap_Sales` (inventory + quote) → review → pay → *(finance → FinTrust partner + DMN)* → `Cap_Warehouse` (pick/QC/dispatch) → `Cap_Loyalty` (accrue) → confirm |
| **Hire** | Check availability → `Cap_Hire` (DMN tiered pricing + booking) → deposit → use → return → `Cap_Warehouse` (inspect) → *(maintenance → FixPro partner + DMN)* → served |

Every journey is orchestrated by one process, `Journey_Customer`. Cancellation paths
(out-of-stock, quote rejected, finance declined, tool unavailable) all reach a clean *Customer
cancelled* end — there are no dead ends.

## 2. Architecture at a glance

- **1 orchestrator** (`Journey_Customer`) — a 3-pool collaboration (Customer + FinTrust + FixPro)
- **4 capability processes** invoked as **call activities**: `Cap_Sales`, `Cap_Warehouse`, `Cap_Loyalty`, `Cap_Hire`
- **2 partner processes** reached by **message correlation**: `Partner_FinTrust`, `Partner_FixPro` (FixPro also has a **timer-triggered** weekly maintenance sweep)
- **4 DMN decision tables**: `finance_assessment`, `loyalty_band`, `hire_pricing`, `maintenance_triage`
- **24 job workers** across seven capability packages + a shared message layer
- **8 H2 tables** recording every side-effect

```
com.probuild.platform
├── PlatformApplication.java      # Spring Boot entry + @Deployment (processes, DMN, forms)
├── shared/     Messages          # single choke point for partner-message publication
├── data/       entity/ repo/ Seeder
├── orchestrator/ OrchestratorHandlers   # journey-init, finance/service request publishers
├── sales/      SalesHandlers
├── warehouse/  WarehouseHandlers
├── loyalty/    LoyaltyHandlers
├── hire/       HireHandlers
├── fintrust/   FinTrustHandlers
└── fixpro/     FixProHandlers
```

## 3. Message correlation channels

| Message | From → To | Correlated on |
|---|---|---|
| `finance.requested` | Customer → FinTrust | `journeyId` |
| `finance.settled` | FinTrust → Customer | `journeyId` |
| `service.requested` | Customer → FixPro | `journeyId` |
| `service.completed` | FixPro → Customer | `journeyId` |

## 4. DMN decision tables

| Decision | Inputs | Outputs |
|---|---|---|
| `finance_assessment` | credit score, income vs amount | recommendation (APPROVE/REFER/DECLINE), term months |
| `loyalty_band` | points balance, has card | discount band, discount % |
| `hire_pricing` | rental days | tier multiplier, tier label |
| `maintenance_triage` | tool count | routine / repair / out-of-service split |

All four are evaluated as **business-rule tasks** inside the processes (visible in the diagrams
with the decision-table icon) — no business rules are hard-coded in Java.

## 5. Build & run

The BPMN files are authored *semantically* in `bpmn-src/` (no diagram layout) and the diagram
interchange is generated by a Node toolchain.

```bash
# 1. generate BPMN diagram layout (single processes + the multi-pool collaboration)
cd tools && npm install && cd ..
node tools/layout.mjs            # bpmn-src/*.bpmn  →  src/main/resources/processes/*.bpmn

# 2. start Camunda 8 (c8run) — Operate/Tasklist at http://localhost:8080 (demo/demo)
./c8run start

# 3. build & run the worker application (port 8081)
mvn package
java -jar target/probuild-platform.jar
```

On startup the app auto-deploys all processes, DMN tables and forms, and the log prints:

```
Deployed: <drd_finance>,<drd_hire>,<drd_loyalty>,<drd_triage>,
          <Cap_Hire>,<Cap_Loyalty>,<Cap_Sales>,<Cap_Warehouse>,
          <Journey_Customer>,<Partner_FinTrust>,<Partner_FixPro>
[seed] tradeCards=3 inventory=11
```

Start a journey in Operate → Processes → `Journey_Customer` → **Start instance**, then complete
user-task forms in Tasklist.

## 6. Toolchain note (why `bpmn-src/`)

`bpmn-auto-layout` lays out single processes; multi-pool collaborations are stitched by
`tools/stitch-collab.mjs`, which lays out each pool's process independently, stacks them as
horizontal pools, and draws the message flows between them. This keeps the source BPMN readable
(pure semantics) while producing fully-rendered, executable diagrams.
